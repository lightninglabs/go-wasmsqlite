Short answer: yes—your plan is sound. The cleanest way to keep using the **unchanged** Go API that sqlc already generated is:

* run the **official SQLite WASM** build in the browser on **OPFS** (ideally the `opfs-sahpool` VFS);
* write a tiny **database/sql driver** in Go (pure Go, using `syscall/js`) which forwards `ExecContext`/`QueryContext`/`QueryRowContext` to a JS/TS bridge living in a Web Worker;
* pass the resulting `*sql.DB` into your sqlc `Queries` as usual.

That approach means your existing sqlc code keeps using the `DBTX`-shaped calls it already emits, which return `*sql.Rows`, `*sql.Row`, `sql.Result`, etc. (you can’t fabricate those without going through `database/sql`)  . The official SQLite WASM docs cover using a VFS with OPFS and the `oo1.DB` constructor (you can pass `vfs: "opfs"` or the higher-perf `"opfs-sahpool"`), and `opfs-sahpool` works broadly without COOP/COEP headers and is currently the fastest persistent option. ([SQLite][1])

Below is a battle-tested architecture and how your **sqlc plugin** can generate the glue.

## Architecture (Go → JS → SQLite WASM on OPFS)

1. **Worker-based JS/TS bridge (generated by the plugin)**

   * Use the ESM package `@sqlite.org/sqlite-wasm`.
   * In the Worker, initialize SQLite and open `/app.db` with the OPFS VFS:

   ```ts
   // bridge.worker.ts (generated)
   import sqlite3InitModule from '@sqlite.org/sqlite-wasm';

   let db: any;

   self.onmessage = async (e) => {
     const { id, type, sql, params, fileName = '/app.db', vfs = 'opfs-sahpool' } = e.data;
     try {
       if (type === 'open') {
         const sqlite3 = await sqlite3InitModule();
         db = new sqlite3.oo1.DB({ filename: fileName, flags: 'ct', vfs });
         postMessage({ id, ok: true });
       } else if (type === 'exec') {
         db.exec(sql, { bind: params ?? [], rowMode: 'array' });
         postMessage({ id, ok: true });
       } else if (type === 'query') {
         const rows = db.selectArrays(sql, params ?? []);
         postMessage({ id, ok: true, rows });
       } else if (type === 'close') {
         db?.close(); db = null;
         postMessage({ id, ok: true });
       }
     } catch (err) {
       postMessage({ id, ok: false, error: String(err) });
     }
   };
   ```

   That uses the official **oo1** API (`sqlite3.oo1.DB`) and its convenience methods like `selectArrays` to keep the wire format simple. ([SQLite][1])
   Tip: Prefer `vfs: 'opfs-sahpool'` for best cross-browser perf and no COOP/COEP requirement. ([SQLite][2])

2. **Go driver (generated by the plugin)**

   * Implement a minimal `database/sql/driver` which talks to the Worker via `syscall/js`.
   * The driver serializes all calls to a single connection (OPFS VFS doesn’t support multi-connection access to the same DB). ([SQLite][2])
   * Map JS values → `driver.Value` (`int64`, `float64`, `bool→int64 0/1`, `string`, `[]byte`, `nil`), and handle context cancellation via `AbortController` + `context.Done()`.

   Sketch:

   ```go
   // wasmsqlite/driver.go (generated)
   package wasmsqlite

   import (
     "context"
     "database/sql"
     "database/sql/driver"
     "errors"
     "syscall/js"
     "time"
   )

   func init() { sql.Register("wasmsqlite", &drv{}) }

   type drv struct{}
   func (d *drv) Open(name string) (driver.Conn, error) { return newConn(name) }

   type conn struct {
     worker js.Value
     reqID  int64
   }

   func newConn(name string) (*conn, error) {
     w := js.Global().Get("Worker").New("bridge.worker.js") // emitted alongside
     c := &conn{worker: w}
     if err := c.callVoid("open", map[string]any{"fileName": name}); err != nil { return nil, err }
     return c, nil
   }

   // ExecContext/QueryContext implement the interfaces used by sqlc.
   func (c *conn) ExecContext(ctx context.Context, query string, args []driver.NamedValue) (driver.Result, error) {
     _, err := c.call("exec", map[string]any{"sql": query, "params": flatten(args)}, ctx)
     return driver.RowsAffected(0), err
   }
   func (c *conn) QueryContext(ctx context.Context, query string, args []driver.NamedValue) (driver.Rows, error) {
     resp, err := c.call("query", map[string]any{"sql": query, "params": flatten(args)}, ctx)
     if err != nil { return nil, err }
     // resp["rows"] is [][]any; wrap into driver.Rows
     return newRows(resp.Get("rows")), nil
   }
   // ... implement Prepare (optional), Begin, Close, and rows iteration.
   ```

   With that, you can do:

   ```go
   // main_wasm.go
   import (
     "database/sql"
     _ "your/plugin/wasmsqlite"
     db "your/sqlc/package"
   )
   func main() {
     sqlDB, _ := sql.Open("wasmsqlite", "/app.db")
     q := db.New(sqlDB) // uses DBTX: ExecContext/QueryContext/QueryRowContext :contentReference[oaicite:6]{index=6}
     // q.CreateAuthor(ctx, ...) etc. :contentReference[oaicite:7]{index=7}
   }
   ```

## What the **sqlc plugin** should emit

You don’t have to replace sqlc’s Go generator. Just **add** a `codegen` block that runs your plugin to drop the bridge files next to your app:

* `bridge.worker.ts` — the Worker shown above.
* `wasmsqlite/driver.go` — the tiny driver skeleton.
* `wasmsqlite/open.go` — helper to open the DB, run `schema.sql` on first open (optional).
* `wasmsqlite/types.go` — (optional) coercion helpers for `sql.NullTime/NullString/NullFloat64` based on the schema.

Your plugin reads sqlc’s **CodegenRequest** (proto), so you already have per-query parameter and column types; you can use that to:

* decide when to coerce text to `time.Time` (your schema has `DATE`/`TIMESTAMP` columns) or numbers to `int64/float64`;
* optionally emit **named prepared statements** for each query so the Worker does less parsing. (But you can start by sending the raw SQL strings—the oo1 API is fast.)

References: sqlc plugin docs + proto definition; the official TypeScript plugin repo, if you want to peek at how they map SQLite types. ([docs.sqlc.dev][3], [buf.build][4], [GitHub][5], [sqlc.dev][6])

## Reusing the **TypeScript plugin**?

You *can*, but you don’t *have* to.

* **Driver route (recommended):** Keep Go the same; generate just the driver + Worker. Lowest effort to integrate with your existing sqlc API.
* **Hybrid route:** Also run `sqlc-gen-typescript` to generate strongly-typed TS wrappers for each query *inside the Worker*, then have the Go driver invoke those named functions instead of sending raw SQL. This can catch type mistakes inside the Worker and makes coercion trivial. (The TS plugin is maintained by sqlc-dev and in Beta. ) ([GitHub][5], [docs.sqlc.dev][7])

## Browser & bundling notes

* Use `@sqlite.org/sqlite-wasm` (the official ESM wrapper) so bundlers pull in the `.wasm` asset automatically. ([GitHub][8])
* Prefer `vfs: 'opfs-sahpool'` for OPFS persistence + performance and no COOP/COEP headers. ([SQLite][2])
* Run SQLite in a **Worker**. OPFS SyncAccessHandle is worker-friendly and the official wrappers support worker usage; it also keeps the main thread snappy. ([SQLite][2])
* One connection per origin/file (serialize access in the driver). ([SQLite][2])

## Gotchas (and how the plugin addresses them)

* **`PrepareContext`, `*sql.Rows`, etc.** — You can’t produce these without `database/sql`. That’s why the driver approach is key; then your existing sqlc `Queries` type “just works.”&#x20;
* **Time values** — Standardize on ISO8601 strings in SQLite and parse to `time.Time` in the driver when sqlc says a column is `DATE`/`TIMESTAMP`. Your schema has both (`published`, `created_at`).&#x20;
* **Context cancellation** — Have the driver wire `context.Context` to an `AbortController` in the Worker and early-return on `ctx.Done()`.
* **Transactions** — Implement `driver.Tx` by sending `BEGIN/COMMIT/ROLLBACK` messages; or use the oo1 `transaction()` helper. ([SQLite][1])

sql:

    queries: query.sql
    codegen:
      - out: internal/db
        plugin: go # the standard Go generator
      - out: webdb
        plugin: your-go-wasm-bridge
